No fitness/health/city overlay was found touching primary class selection anywhere in supabase/functions/.

2. Add-on structure & the segment-vocabulary gap

household_members.segment CHECK-constrained to 8 UPPER_SNAKE values: INFANT, TODDLER, SCHOOL_CHILD, DIABETIC_ELDER, POSTPARTUM, FITNESS_OVERLAY, FASTING_MEMBER, ADULT_STANDARD
re_addon_classes.segment / re_household_addon_plans.segment plain text, NO CHECK constraint, NO FK — 19 distinct live lower_snake_case values: allergy_member, baby_6_18m, child_or_picky_child, cook_needs_instruction, diabetic_member, elderly_member, fasting_member, gym_high_protein_member, hypertension_heart_member, jain_member, lactating_or_postpartum_mother, picky_child, postpartum_mother, pregnant_member, recovery_member, school_child, teen_high_appetite, toddler, weight_loss_member, working_kitchen_manager
GAP — verified live: no FK, CHECK, or shared value exists between these two vocabularies (case mismatch alone breaks equality: TODDLER vs toddler). No mapping/translation table exists anywhere in public or re_engine.
GAP — verified in code: supabase/functions/_shared/services/re/types.ts's HouseholdMember interface carries only segment / allergenFlags / isActive — no dietType field. supabase-stores.ts:persistHouseholdMembers writes only profile_id, member_name, segment, allergen_flags — it never writes household_members.diet_type. A grep for "addon" across all of supabase/functions/ returns exactly 2 hits, both comments stating add-ons are "separate structures" not yet built — not one line of runtime addon-generation logic exists.
public.addon_slots — 0 live rows. re_household_addon_plans — 7,992 rows (cohort-level static config, not per-household instance data: cohort_id + segment → addon_class_code).
